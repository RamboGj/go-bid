# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`go-bid` is a real-time auction backend in Go. Users sign up, create product auctions, and
connect over WebSocket to place bids that are broadcast live to everyone watching that auction.

## Commands

The DB connection for every command is read from `.env` (loaded via `godotenv`). Start Postgres first.

```sh
docker compose up -d                              # start Postgres 17 (uses GOBID_DATABASE_* from .env)
make run-api                                      # run the API with hot reload via air (also `air`)
go run ./cmd/api                                  # run the API without hot reload (localhost:3080)

# Migrations (tern) — both forms are equivalent:
go run ./cmd/terndotenv                           # loads .env, then runs tern migrate
make migrate                                      # runs tern directly (env must already be exported)

sqlc generate -f ./internal/store/pgstore/sqlc.yml   # regenerate Go from SQL after editing queries/ or migrations/

go test ./...                                      # run all tests
go test ./internal/services -run TestName          # run a single test
```

External tools required on PATH: `tern` (migrations), `sqlc` (codegen), `air` (hot reload).

## Architecture

Layered, with dependencies pointing inward: `cmd → api → services → store/pgstore`.

- **`cmd/api`** — composition root. `main.go` builds the `pgxpool`, the `scs` session
  manager (Postgres-backed via `pgxstore`), the `websocket.Upgrader`, and a single
  `services.AuctionLobby`, injecting them all into one `api.Api` struct. Routes are bound in
  `internal/api/routes.go`.

- **`cmd/terndotenv`** — wrapper that loads `.env` then shells out to `tern migrate`. Exists
  because `tern.conf` references `GOBID_DATABASE_*` env vars that `.env` provides.

- **`internal/api`** — HTTP layer (chi router). The `Api` struct (`api.go`) holds all services
  and is the receiver for every handler. Auth is session-cookie based: `AuthMiddleware` checks
  for `"AuthenticatedUserId"` in the scs session; handlers read the user id back out with
  `api.Sessions.Get(ctx, "AuthenticatedUserId").(uuid.UUID)`.

- **`internal/usecase/<entity>`** — request DTOs that implement `validator.Validator`
  (a `Valid(ctx) Evaluator` method). They carry validation rules only, no business logic.

- **`internal/services`** — business logic, one service per entity wrapping the `pgstore`
  `Queries`. `NewXxxService(pool)` constructors. Services define their own sentinel errors
  (e.g. `ErrProductNotFound`, `ErrBidIsTooLow`) that handlers match with `errors.Is`.

- **`internal/store/pgstore`** — **generated by sqlc; do not hand-edit `*.sql.go`, `models.go`,
  or `db.go`.** Edit the SQL in `queries/` and the schema in `migrations/`, then rerun `sqlc generate`.
  UUID columns map to `github.com/google/uuid.UUID` and `timestamptz` to `time.Time` (see `sqlc.yml`).

### Request/response convention

All JSON I/O goes through `internal/jsonutils`. Handlers decode+validate in one step with
`jsonutils.DecodeValidJson[T](r)` which returns `(data, problems, err)`: when `problems != nil`
respond `422` with the field-error map, otherwise `400`. Write responses with
`jsonutils.EncodeJson(w, r, status, payload)`. The `Evaluator` (a `map[string]string` of
field→message in `internal/validator`) keeps the **first** error per field.

### Real-time auctions (the core of the system)

This is the part that spans multiple files and is non-obvious:

- Creating a product (`handleCreateProduct`) spins up an `AuctionRoom` whose `context` has a
  deadline set to the product's `AuctionEnd`, starts `room.Run()` in a goroutine, and registers
  it in the `AuctionLobby.Rooms` map (guarded by the lobby mutex). When the deadline fires,
  `Run()` broadcasts `AuctionFinished` to all clients and returns.

- `AuctionRoom.Run()` is a single-goroutine event loop (`auction_service.go`) owning the room
  state. It serializes everything over channels — `Register`, `Unregister`, `Broadcast`, and
  `Context.Done()` — so the `Clients` map needs no locking.

- A WebSocket subscriber (`handleSubscribeUserToAuction`) looks up the room, upgrades the
  connection, and starts two goroutines per client: `ReadEventLoop` (socket → `room.Broadcast`)
  and `WriteEventLoop` (`client.Send` → socket, plus periodic pings). **Each goroutine closes
  the connection in its own defer; the handler must not close it.**

- Messages are typed by the `MessageKind` iota (`PlaceBid`, `SuccessfullyPlacedBid`,
  `FailedToPlaceBid`, `InvalidJSON`, `NewBidPlaced`, `AuctionFinished`). On `PlaceBid` the room
  calls `BidsService.PlaceBid`; on success it echoes `SuccessfullyPlacedBid` to the bidder and
  `NewBidPlaced` to everyone else; `ErrBidIsTooLow` becomes a `FailedToPlaceBid` to just the bidder.

## Notes

- CSRF (`gorilla/csrf`) is wired up but **commented out** in `routes.go` for development; the
  `GOBID_CSRF_KEY` env var and `HandleGetCSRFToken` exist for when it's re-enabled.
- The committed `.env` contains local dev credentials only.
